Algolia short-lived key + 429 demo

This demo runs two simulated users. The real user receives a short-lived secured API key and searches normally. The bot also receives one short-lived secured API key, but the backend acts like an application, CDN, or WAF layer: it detects the bot fingerprint and returns Backend 429s before those requests are allowed to reach Algolia.

After the Backend 429 phase, the demo intentionally lets the remaining bot traffic continue to Algolia using the same short-lived secured API key. This shows Algolia acting as the second layer of protection: once the secured key exceeds the parent key's low API-call limit, Algolia returns Algolia 429s.
